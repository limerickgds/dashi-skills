# 现代 AI Agent 框架的控制论系统分析

> 本文件从控制论视角对当前主流 AI Agent 框架进行系统分析，识别其反馈结构、稳定性特征、控制模式和潜在故障机理。

---

## 一、分析框架：控制论视角的 Agent 评估维度

### 1.1 评估维度定义

| 维度 | 控制论概念 | Agent 中的对应 | 评价标准 |
|------|-----------|---------------|---------|
| 反馈结构 | 反馈回路类型与层级 | Agent 如何获取和利用执行结果 | 闭环/开环、反馈层数、延迟 |
| 稳定性 | 系统收敛/发散行为 | Agent 是否收敛到正确答案或陷入循环 | 收敛速度、振荡幅度、终止条件 |
| 多样性匹配 | Ashby 必要多样性定律 | 工具和策略是否匹配任务复杂度 | 工具集丰富度、策略灵活性 |
| 可观测性 | 状态观测能力 | Agent 对自身状态和环境的感知 | 中间状态可见性、错误可检测性 |
| 可控性 | 系统状态可操纵性 | 人类对 Agent 行为的干预能力 | 可打断性、方向可调整性 |
| 信息效率 | 信息利用率 | Agent 对上下文信息的利用效率 | 冗余度、信噪比、信息压缩 |

## 二、ReAct 框架的控制论解读

### 2.1 框架概述

ReAct (Reasoning + Acting, Yao et al., 2023) 将推理链（Chain of Thought）与工具调用交替进行，形成 Thought → Action → Observation 循环。

### 2.2 控制论结构分析

```
                    反馈闭环
                 ┌─────────────────────────────┐
                 │                             │
[Task/Goal] → [Thought] → [Action] → [Observation] → [Thought] → ...
                 │ 推理环节   │ 执行环节    │ 感知环节
                 │           │             │
                 └─ 控制器 ──┘── 执行器 ───┘── 传感器
```

#### 反馈闭环特征

| 控制论要素 | ReAct 对应 | 分析 |
|-----------|-----------|------|
| 参考输入(r) | 用户任务/目标 | 通常为自然语言描述的任务 |
| 控制器 | LLM Thought 步骤 | 基于当前状态推理下一步行动 |
| 执行器 | Action（工具调用） | 执行搜索、计算、API调用等 |
| 传感器 | Observation | 获取工具执行的返回结果 |
| 误差信号 | 隐式的"任务是否完成"判断 | LLM 在 Thought 中评估进展 |

#### 误差修正机制

ReAct 的误差修正是**隐式的**：
1. LLM 在 Thought 中对比当前信息与目标需求。
2. 如果信息不足或不正确，生成新的 Action 来获取更多/更准确的信息。
3. 这是一种**零阶反馈**——直接对当前观察做出响应，不做预测。

### 2.3 适应性分析

**优势**：
- 每一步都有真实环境反馈（Observation），避免了纯推理的幻觉积累。
- 交替推理和行动使得策略可以动态调整。

**局限**：
- **无元反馈**：Agent 不反思自身推理策略的有效性，属于单层反馈。
- **无记忆跨 episode**：每次任务独立，不从历史任务中学习。
- **反馈延迟**：每步的 Observation 只反映该步的结果，对全局进展的评估依赖 LLM 能力。

### 2.4 控制模式分类

ReAct 属于**跟踪控制**——持续监测当前状态与目标的偏差，动态调整行动，但不修改自身的控制策略。

## 三、Reflexion 框架的二阶反馈特征

### 3.1 框架概述

Reflexion (Shinn et al., 2023) 在 ReAct 的基础上增加了"自我反思"机制：Agent 在任务失败后对自己的失败原因进行分析，并将反思结论存入记忆以指导下次尝试。

### 3.2 二阶反馈结构

```
第一阶反馈（行动层 —— ReAct 循环）：
  Thought → Action → Observation → Thought → ... → 任务结果

第二阶反馈（反思层）：
  任务结果 → [Evaluator] → 成功/失败判定
                               │
                               ↓（失败时触发）
                         [Self-Reflection]
                               │
                               ↓
                     [Reflexion Memory] ←── 存储反思结论
                               │
                               ↓（下一次尝试时读取）
                     新的 ReAct 循环（携带反思记忆）
```

### 3.3 与 Ashby 超稳定系统的对应

| Ashby 超稳定系统 | Reflexion |
|-----------------|-----------|
| 第一层反馈（快速、连续） | ReAct 的 Thought-Action-Observation 循环 |
| 第二层反馈（慢速、离散） | 任务失败后的 Self-Reflection |
| 参数随机重置 | 反思结论改变下一次尝试的策略 |
| 阈值触发 | 任务失败触发反思 |
| 试错搜索 | 多次尝试直到成功或达到最大尝试次数 |

### 3.4 控制论优势

1. **显式误差分析**：不仅知道"失败了"，还分析"为什么失败"。
2. **策略调整**：根据反思修改后续的推理和行动策略。
3. **经验积累**：反思记忆跨尝试保留，避免重复同样的错误。

### 3.5 控制论局限

1. **反思质量依赖 LLM**：如果 LLM 的反思分析不准确，会引入错误的"学习"。
2. **只有失败触发**：成功时不反思，可能错过优化机会。
3. **记忆容量有限**：反思记忆不断增长，最终受上下文窗口限制。
4. **无参数级适应**：不改变 LLM 自身的参数，只在 prompt 层面调整。

### 3.6 控制模式分类

Reflexion 属于**自适应控制**的初级形式——具有双层反馈和策略调整能力，但适应范围受限于上下文记忆。

## 四、AutoGPT 的自主循环与正反馈失控

### 4.1 框架概述

AutoGPT (Richards, 2023) 是最早的自主 Agent 之一：给定一个高级目标，Agent 自动分解子目标、执行行动、评估进展，循环运行直到完成。

### 4.2 循环结构

```
[高级目标] → [目标分解] → [优先级排序] → [执行最高优先级任务]
                  ↑                              │
                  │                              ↓
                  │                     [评估执行结果]
                  │                              │
                  └──── [更新目标列表] ←──────────┘
```

### 4.3 正反馈失控问题

AutoGPT 的核心控制论问题是**正反馈失控**：

#### 失控机制

1. **目标膨胀**：Agent 在执行过程中发现新的"相关"子目标，将其加入任务列表。
2. **缺乏有效的负反馈**：Agent 难以判断哪些子目标真正有助于原始目标。
3. **目标列表无限增长**：新增的子目标又会衍生出更多子目标，形成正反馈循环。
4. **偏离原始目标**：Agent 越来越深入地追逐次要目标，忘记了原始任务。

#### 控制论诊断

```
正反馈回路：
[执行任务] → [发现新问题] → [创建新子目标] → [任务列表膨胀] → [执行更多任务] → ...
                                                                        ↑          │
                                                                        └──────────┘
缺失的负反馈：
[原始目标] ←?→ [当前活动] 之间缺乏有效的偏差评估机制
```

#### 类比：正反馈失控的经典案例

| 系统 | 正反馈机制 | 失控表现 | 缺失的负反馈 |
|------|-----------|---------|-------------|
| AutoGPT | 子目标自我繁殖 | 目标漂移、资源耗尽 | 目标相关性评估 |
| 麦克风-扬声器 | 声音循环放大 | 啸叫 | 音量限制器 |
| 股市泡沫 | 上涨 → 买入 → 更多上涨 | 崩盘 | 理性估值 |
| 核裂变 | 中子 → 裂变 → 更多中子 | 爆炸 | 控制棒吸收中子 |

### 4.4 改进方向

1. **引入硬性约束**：设定最大子目标数量和执行步数。
2. **目标相关性评估**：每个新子目标必须通过与原始目标的相关性检验。
3. **资源预算**：设定 token/时间/工具调用次数的预算上限。
4. **人类监督检查点**：在关键决策点引入人类确认。

## 五、LangGraph 的状态空间方法

### 5.1 框架概述

LangGraph (LangChain, 2024) 将 Agent 的执行过程建模为一个有向图（状态机），节点是处理步骤，边是条件转移。

### 5.2 状态空间表示

LangGraph 显式地使用了控制论中的**状态空间方法**：

```python
# LangGraph 的状态空间结构
class AgentState(TypedDict):
    messages: list       # 对话历史（状态变量）
    plan: list           # 当前计划（状态变量）
    results: dict        # 执行结果（状态变量）
    iteration: int       # 迭代次数（状态变量）

# 状态转移函数
def node_function(state: AgentState) -> AgentState:
    # x(t+1) = f(x(t), u(t))
    new_state = process(state)
    return new_state
```

### 5.3 控制论分析

#### 优势

1. **可观测性**：所有状态变量显式定义，系统状态完全可观测。
2. **可控性**：可以在任何节点插入人类干预、条件检查、循环限制。
3. **确定性**：状态转移规则明确，行为可预测。
4. **可检查点**：支持状态的存储和恢复（checkpoint），实现"时间旅行"调试。

#### 与控制论的精确对应

| 控制论概念 | LangGraph 实现 |
|-----------|---------------|
| 状态空间 x(t) | TypedDict 状态对象 |
| 状态转移 f(x,u) | 节点函数 |
| 输入 u(t) | 条件边的触发条件 |
| 输出 y(t) | 最终节点的输出 |
| 反馈 | 带条件的回边（循环） |
| 稳定性约束 | max_iterations、终止条件 |

#### 局限

1. **需要预先设计图结构**：灵活性不如自由形式的 Agent。
2. **状态空间必须预定义**：难以处理完全未预见的信息类型。
3. **图的复杂度**：随着需求增加，图结构可能变得极其复杂。

## 六、Voyager 的技能库正反馈机制

### 6.1 框架概述

Voyager (Wang et al., 2023) 是 Minecraft 游戏中的自主 Agent，核心创新是维护一个**技能库**——Agent 编写的代码函数被存储起来供后续复用。

### 6.2 正反馈机制分析

```
[新任务] → [尝试用已有技能解决]
              │
              ├── 成功 → 技能复用 → 更快完成类似任务
              │                        │
              │                        ↓ (正反馈：成功强化)
              │                   技能库质量提升
              │
              └── 失败 → [LLM 编写新技能] → [环境验证]
                                                │
                              ├── 验证成功 → [加入技能库] → 能力扩展
                              │                              │
                              │                              ↓ (正反馈：能力增长)
                              │                        可解决更多任务
                              │
                              └── 验证失败 → [迭代改进代码]
```

### 6.3 控制论特征

#### 受控的正反馈

与 AutoGPT 的失控正反馈不同，Voyager 的正反馈是**受控的**：

| 维度 | AutoGPT | Voyager |
|------|---------|---------|
| 正反馈对象 | 目标列表（无限增长） | 技能库（有质量门槛） |
| 质量控制 | 缺乏 | 环境验证（代码必须运行成功） |
| 负反馈约束 | 弱 | 强（验证失败的代码不入库） |
| 增长方向 | 发散（目标漂移） | 收敛（能力单调增长） |
| 稳定性 | 不稳定 | 稳定的能力积累 |

#### 环境验证作为负反馈

- 技能代码必须在 Minecraft 环境中**实际执行成功**才会被保留。
- 这是一种**硬约束负反馈**——不是 LLM 的主观判断，而是客观的环境验证。
- 这防止了幻觉代码进入技能库。

## 七、Godel Agent 的自指框架与二阶控制论

### 7.1 框架概述

Godel Agent (2024) 受哥德尔自指定理启发，设计了一个能修改自身 Prompt 和策略的 Agent 框架。Agent 不仅执行任务，还能反思和修改自己的"操作系统"（system prompt、工具选择策略、推理模板等）。

### 7.2 自指结构

```
┌─────────────────────────────────────────┐
│          Godel Agent 自指架构            │
│                                         │
│  ┌──────────────────┐                   │
│  │  Meta-Level      │ ← 二阶控制器      │
│  │  (自我修改策略)   │                   │
│  └────────┬─────────┘                   │
│           │ 修改                         │
│           ↓                              │
│  ┌──────────────────┐                   │
│  │  Object-Level    │ ← 一阶控制器      │
│  │  (任务执行策略)   │                   │
│  └────────┬─────────┘                   │
│           │ 执行                         │
│           ↓                              │
│  ┌──────────────────┐                   │
│  │  Environment     │ ← 被控对象        │
│  │  (任务环境)       │                   │
│  └──────────────────┘                   │
└─────────────────────────────────────────┘
```

### 7.3 二阶控制论分析

#### 与二阶控制论的对应

二阶控制论（Second-Order Cybernetics, von Foerster）的核心命题是：**观察者本身也是系统的一部分**。

| 二阶控制论概念 | Godel Agent 实现 |
|--------------|-----------------|
| 观察者嵌入系统 | Agent 反思自身的推理过程 |
| 自我指涉(self-reference) | Agent 修改自身的 Prompt |
| 递归认识论 | 元策略指导对象策略，对象策略结果反馈给元策略 |
| 自创生(autopoiesis) | Agent 通过自我修改"创造"新版本的自己 |

#### 潜在风险：自指悖论

- **哥德尔不完备性**：一个足够复杂的系统不能完全证明自身的一致性。
- **Agent 风险**：Agent 修改自身时，如何保证修改不会破坏自身的核心能力？
- **控制论对策**：需要"不可修改的核心"（invariant core）作为稳定锚点。

## 八、Multi-Agent 系统的分层控制

### 8.1 CrewAI 的层级结构

```
[Manager Agent] ← 战略层控制器
      │
      ├── [Researcher Agent] ← 执行层控制器 1
      ├── [Writer Agent]     ← 执行层控制器 2
      └── [Reviewer Agent]   ← 执行层控制器 3（质量反馈）
```

#### 控制论分析

- **分层控制**：Manager 设定目标和分配任务，Worker 执行具体操作。
- **反馈路径**：Worker 的输出反馈给 Manager，Manager 根据整体进展调整任务分配。
- **Reviewer 作为负反馈**：专门的审查角色提供质量反馈，增强系统的误差检测能力。

### 8.2 AutoGen 的对话式多 Agent

```
[Agent A] ←→ [Agent B] ←→ [Agent C]
    对话式交互，无固定层级
```

#### 控制论特征

| 特征 | 分析 |
|------|------|
| 控制结构 | 分布式，无中心控制器 |
| 反馈机制 | Agent 间的对话本身就是反馈 |
| 稳定性 | 依赖对话收敛性，可能出现无限对话循环 |
| 多样性匹配 | 多个 Agent 的知识/能力互补，增加系统多样性 |
| 风险 | 幻觉共振——Agent 之间互相强化错误信息 |

### 8.3 Multi-Agent 的控制论对比

| 维度 | CrewAI (层级式) | AutoGen (对话式) |
|------|----------------|-----------------|
| 控制结构 | 集中式层级控制 | 分布式对等协商 |
| 反馈速度 | 较慢（经过Manager） | 较快（直接对话） |
| 稳定性 | 较高（Manager约束） | 较低（可能发散） |
| 灵活性 | 较低（固定角色） | 较高（动态角色） |
| 瓶颈 | Manager 能力上限 | 共识达成困难 |
| 控制论类比 | 军事指挥链 | 市场协商机制 |

## 九、SiriuS 的自举推理

### 9.1 框架概述

SiriuS 采用自举（bootstrapping）方法构建 Agent 推理能力：从简单任务开始，逐步用已解决的简单任务的经验来指导更复杂任务的解决。

### 9.2 控制论分析

```
Level 0: [简单任务] → [基本策略] → [解决]
                                      │
                                      ↓ 经验提取
Level 1: [中等任务] + [Level 0 经验] → [改进策略] → [解决]
                                                      │
                                                      ↓ 经验提取
Level 2: [复杂任务] + [Level 0+1 经验] → [高级策略] → [解决]
```

#### 正反馈的阶梯式增长

- 每一层的成功经验为下一层提供"脚手架"。
- 这是一种**受约束的正反馈**：能力增长是阶梯式的，每一步都有验证。
- 与 Voyager 的技能库正反馈类似，但更强调**难度渐进**。

#### 与超稳定系统的关系

- Level 0 对应第一层稳态（在简单任务上稳定）。
- 进入 Level 1 相当于"扰动超过阈值"——现有策略不够用了。
- 利用 Level 0 经验进行"参数调整"——不是随机重置，而是基于经验的定向调整。
- 这比 Ashby 的随机试错更高效，是**有指导的超稳定适应**。

## 十、OODA Loop 和 PDCA Cycle 在 Agent 中的映射

### 10.1 OODA Loop (Observe-Orient-Decide-Act)

```
[Observe] → [Orient] → [Decide] → [Act]
    ↑                                 │
    └─────── 反馈 ────────────────────┘
```

#### Agent 映射

| OODA 阶段 | Agent 对应 | 控制论角色 |
|-----------|-----------|-----------|
| Observe（观察） | 获取工具输出、用户输入 | 传感器 |
| Orient（定向） | 上下文分析、情况评估 | 状态估计器 |
| Decide（决策） | 选择下一步行动 | 控制器 |
| Act（行动） | 执行工具调用 | 执行器 |

#### 控制论特征

- OODA 强调**速度**——决策循环越快，越能获得优势（Boyd 的"时间竞争"概念）。
- Orient 是 OODA 的核心——它包含了心智模型、文化因素、经验积累，决定了观察如何被解读。
- 在 Agent 中，Orient 对应 LLM 的"理解"能力——如何将原始观察转化为有意义的状态评估。

### 10.2 PDCA Cycle (Plan-Do-Check-Act)

```
[Plan] → [Do] → [Check] → [Act(Adjust)]
   ↑                           │
   └────── 持续改进 ────────────┘
```

#### Agent 映射

| PDCA 阶段 | Agent 对应 | 控制论角色 |
|-----------|-----------|-----------|
| Plan（计划） | 任务分解、策略制定 | 前馈控制 |
| Do（执行） | 工具调用、行动执行 | 执行 |
| Check（检查） | 结果验证、目标比对 | 反馈检测 |
| Act（改进） | 策略调整、错误修正 | 控制调整 |

### 10.3 OODA vs PDCA vs ReAct 对照

| 维度 | OODA | PDCA | ReAct |
|------|------|------|-------|
| 起源 | 军事决策 | 质量管理 | AI Agent |
| 核心强调 | 速度与适应 | 持续改进 | 推理与行动交替 |
| 反馈形式 | 隐式（重新观察） | 显式（Check阶段） | 显式（Observation） |
| 规划 | 隐式（Orient中） | 显式（Plan阶段） | 隐式（Thought中） |
| 适合场景 | 快速变化环境 | 稳定改进过程 | 信息获取型任务 |

## 十一、Agent 自主性分级 (L1-L5)

### 11.1 分级框架

借鉴自动驾驶的分级体系，从控制论角度定义 Agent 自主性等级：

| 等级 | 名称 | 控制模式 | 人类角色 | 反馈结构 | 代表框架 |
|------|------|---------|---------|---------|---------|
| L1 | 辅助执行 | 程序控制 | 每步确认 | 人在环中的开环 | 简单 Prompt + 单次 LLM 调用 |
| L2 | 部分自主 | 跟踪控制 | 关键节点审批 | 单层反馈 + 人类检查点 | ReAct（带人类确认） |
| L3 | 条件自主 | 跟踪控制 + 异常处理 | 异常时介入 | 单层反馈 + 异常检测 | ReAct + 错误处理逻辑 |
| L4 | 高度自主 | 自适应控制 | 目标设定与验收 | 多层反馈 | Reflexion, LangGraph 复杂图 |
| L5 | 完全自主 | 自组织 | 仅观察 | 自指反馈 + 元学习 | Godel Agent (理论上) |

### 11.2 各等级的控制论特征详解

#### L1：辅助执行

```
[人类] → [指令] → [Agent 单步执行] → [结果] → [人类判断] → [下一步指令]
```
- 控制器：人类
- Agent：纯执行器
- 反馈：人类判断

#### L2：部分自主

```
[人类设定目标] → [Agent 多步执行] → [关键点请求人类确认] → [继续/调整]
```
- 控制器：Agent + 人类（混合控制）
- 反馈：Agent 的自动反馈 + 人类在检查点的审核反馈

#### L3：条件自主

```
[人类设定目标] → [Agent 自主执行] ── 正常路径 ──→ [完成]
                                    │
                                    └── 异常检测 → [请求人类介入]
```
- 控制器：Agent（正常时）/ 人类（异常时）
- 新增：异常检测机制（超出置信度阈值时请求帮助）

#### L4：高度自主

```
[人类设定目标] → [Agent 自主执行 + 自我反思 + 策略调整] → [交付结果]
```
- 控制器：Agent（含元控制层）
- 人类：验收者

#### L5：完全自主

```
[Agent 自主发现问题 + 设定目标 + 执行 + 反思 + 自我改进]
人类：观察者/合作伙伴
```
- 控制器：Agent（含自指能力）
- 目前尚无可靠实现

## 十二、常见 Agent 故障的控制论诊断

### 12.1 故障诊断速查表

| 故障现象 | 控制论诊断 | 根本原因 | 修复策略 |
|---------|-----------|---------|---------|
| 目标漂移 | 正反馈失控 | 缺乏与原始目标的持续比对 | 增加目标相关性检查的负反馈 |
| 幻觉传播 | 错误信息的正反馈放大 | 缺乏外部验证（传感器失灵） | 引入事实验证环节（环境负反馈） |
| 无限循环 | 反馈回路缺乏收敛条件 | 终止条件不足或误差信号无变化 | 添加最大迭代限制和进展检测 |
| 输出振荡 | 反馈增益过高 + 延迟 | 每步修正过度，来回摆动 | 降低修正幅度、增加阻尼机制 |
| 过度保守 | 负反馈过强 | 过度的安全检查抑制了行动 | 调整反馈增益，放宽安全阈值 |
| 能力不足 | 多样性不匹配 | 工具集/知识不足以应对任务复杂度 | 增加工具或缩小任务范围(Ashby定律) |
| 上下文溢出 | 信道容量超限 | 信息量超过上下文窗口容量 | 信息压缩、摘要、选择性遗忘 |
| 决策瘫痪 | 控制器过载 | 选项过多、评估标准冲突 | 简化决策空间、引入启发式规则 |
| 错误累积 | 开环或弱反馈 | 中间步骤无验证 | 增加中间检查点 |
| 灾难性遗忘 | 状态信息丢失 | 长程任务中早期信息被遗忘 | 关键信息持久化存储 |

### 12.2 详细故障分析

#### 故障 1：目标漂移（Goal Drift）

**控制论模型**：
```
期望目标 G₀ → [Agent 执行] → 实际目标逐渐偏移 → G₁ → G₂ → ... → Gₙ ≠ G₀

原因：每一步的行动略微偏离目标方向，但缺乏与 G₀ 的持续比对（反馈断裂）
```

**诊断指标**：
- 后续行动与原始目标的语义相似度逐步下降。
- Agent 开始关注与原始任务无关的子问题。

**修复机制**：
```python
# 目标锚定反馈
for each step:
    relevance = evaluate_relevance(current_action, original_goal)
    if relevance < THRESHOLD:
        redirect_to_original_goal()
```

#### 故障 2：幻觉传播（Hallucination Propagation）

**控制论模型**：
```
[LLM 生成幻觉信息 H] → [H 进入上下文] → [后续推理基于 H] → [更多幻觉]
正反馈回路：幻觉 → 被当作事实 → 产生更多基于幻觉的推理
缺失的负反馈：缺乏对 H 的事实验证
```

**在 Multi-Agent 中尤其危险**：
- Agent A 产生幻觉 → 传递给 Agent B → Agent B 基于幻觉推理 → 回传给 Agent A → 幻觉被"确认"。
- 这是典型的正反馈共振现象。

#### 故障 3：无限循环（Infinite Loop）

**控制论模型**：
```
[状态 S₁] → [行动 A] → [状态 S₂] → [行动 B] → [状态 S₁] → [行动 A] → ...
```

**根本原因**：系统在状态空间中进入了一个**极限环（limit cycle）**——反馈回路存在，但不收敛到固定点，而是在几个状态之间循环。

**诊断方法**：检测连续 N 步的状态/行动序列是否出现重复模式。

#### 故障 4：输出振荡（Output Oscillation）

**控制论模型**：
```
[写代码] → [测试失败：缺少功能A] → [添加功能A] → [测试失败：功能A破坏了B]
→ [修复B] → [测试失败：修复B又破坏了A] → ...
```

**类比**：这与维纳描述的小脑震颤是同一类现象——反馈修正过度，导致系统在目标两侧来回振荡。

**修复**：
- 降低每次修正的幅度（增加"阻尼"）。
- 引入更全面的影响评估（在修改 A 时预判对 B 的影响）。
- 限制修改的范围和频率。

### 12.3 综合诊断流程图

```
Agent 出现异常行为
    │
    ├── 行为不收敛？
    │      ├── 重复循环 → 诊断：极限环 → 添加循环检测和终止条件
    │      └── 持续发散 → 诊断：正反馈失控 → 增加负反馈约束
    │
    ├── 输出质量下降？
    │      ├── 逐步退化 → 诊断：错误累积/幻觉传播 → 增加中间验证
    │      └── 来回波动 → 诊断：振荡 → 增加阻尼、降低修正增益
    │
    ├── 偏离目标？
    │      ├── 渐进偏离 → 诊断：目标漂移 → 增加目标锚定反馈
    │      └── 突然偏离 → 诊断：上下文溢出/关键信息丢失 → 信息持久化
    │
    └── 无法完成任务？
           ├── 能力不足 → 诊断：多样性不匹配 → 增加工具或缩小任务
           └── 决策瘫痪 → 诊断：控制器过载 → 简化决策空间
```
